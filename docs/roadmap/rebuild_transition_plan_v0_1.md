# Junior Study Trainer 完成版再構築への移行計画 v0.1

## 1. この文書の目的

Junior Study Trainer は、数か月にわたり試作・実運用を重ね、
学習機能、教材管理、進捗管理、目的別問題集、弱点・定着管理などの要求を確認してきた。

現行アプリは要求確認用の試作として十分な役割を果たしたため、
今後は現行構造への機能追加を終了し、
完成版アーキテクチャ上で新たに構築する。

この文書は、現行試作を安全に保存し、
完成版の再開発を混乱なく開始するための移行方針を定める。

## 2. 基本判断

今後は次の方針を採用する。

- 現行アプリを完成版へ段階的に改修しない。
- 現行アプリのコード構造を完成版の制約にしない。
- 現行の問題データ形式を完成版の互換対象にしない。
- 現行の履歴・進捗・バックアップデータは移行しない。
- 旧形式との後方互換機能は原則として実装しない。
- これまでに確認した要求、設計知識、教材内容は参考資産として残す。
- 完成版はサーバー、DB、メディア対応を前提として設計する。
- 最短で完成へ到達するため、完成版の構造上で縦方向に機能を通す。

## 3. リポジトリ方針

同じ GitHub リポジトリを継続して使用する。

現行試作は、次の方法で凍結保存する。

- 保存用ブランチ: `archive/prototype-v0.6`
- 保存用タグ: `prototype-v0.6-final`

凍結後の `main` は、完成版の正式開発ブランチとする。

旧版を `legacy/` などのフォルダとして
完成版の作業ツリー内へ残さない。

旧コードや旧設計を確認する必要がある場合は、
保存用ブランチ、タグ、または Git 履歴を参照する。

## 4. 現行試作から引き継がないもの

次は完成版へ直接移行しない。

- `app/index.html` を中心とした現行アプリ構造
- 現行の JavaScript 問題データ形式
- localStorage の保存キーと保存形式
- バックアップ JSON version 1～6 の互換処理
- 既存の回答履歴
- 既存の問題別進捗
- 既存の目的別進捗
- 既存の周回状態
- 既存の弱点・定着派生状態
- GitHub Pages のみで完結する構成
- 現行アプリ向けの段階的リファクタリング計画
- 旧問題データの移行を前提とした計画

## 6. 完成版の前提

完成版は、少なくとも次を前提として設計する。

- Web アプリ
- サーバー API
- データベース
- 問題・教材・カタログの永続管理
- 回答履歴の永続管理
- 画像・音声などのアセット管理
- 複数端末利用
- 将来の認証対応
- 問題データのスキーマ検証
- 自動テスト可能な構造
- UI、ドメインロジック、保存処理の分離

ただし、初期段階から過剰な分散構成や
マイクロサービス構成にはしない。

単一 Web アプリ、単一 API、単一 DB を基本とする。

## 7. 最短で進めるための開発原則

完成版を一括で全機能実装しない。

完成版アーキテクチャ上で、次の縦方向の経路を先に完成させる。

1. 学習内容カタログを登録する。
2. 新方式の問題を登録する。
3. 問題を検索・取得する。
4. 子ども用画面へ表示する。
5. 回答・判定を記録する。
6. 回答履歴を DB へ保存する。
7. 問題別進捗と定着状態を計算する。
8. 画像・表・グラフ・音声を表示する。
9. 問題・解答・解説を印刷出力する。
10. 管理画面から登録内容を確認する。

この経路が通った後に、
教材数、問題数、目的別問題集、分析機能を拡張する。

## 8. 新方式問題データの作成開始時期

新方式問題データの大量作成は、
次が確定するまで開始しない。

- 完成版の技術構成
- 問題データスキーマ
- DB の主要テーブル
- カタログ参照方式
- 問題、解答、解説の表現方式
- 画像・音声アセットの参照方式
- ID・コード体系
- バリデーション方式

上記の初版が確定した時点で、
基準問題セットを作成する。

基準問題セットには、少なくとも次を含める。

- テキスト一問一答
- 数式
- 画像付き問題
- 表の読み取り
- グラフの読み取り
- 長文・会話文
- 音声再生
- 記述式
- 複数正解
- 詳細解説
- 印刷対象問題

各形式を少数ずつ作成し、
合計20～30問程度を目安とする。

基準問題セットが、
登録、表示、回答、履歴保存、進捗計算、印刷まで通った後に、
正式教材の本格作成を開始する。

## 9. 完成版再開発の初期工程

### フェーズ0: 現行試作の凍結

- この移行計画を確定する。
- 現行状態をコミットする。
- `archive/prototype-v0.6` ブランチを作成する。
- `prototype-v0.6-final` タグを付ける。
- GitHub へブランチとタグを push する。

### フェーズ1: 完成版要求の統合

- 既存設計書を要求単位で棚卸しする。
- 現行実装依存の記述を除外する。
- 完成版の機能要求を1つの文書へ統合する。
- 必須、初期リリース後、将来候補に分類する。

### フェーズ2: 技術構成の決定

- Web フレームワーク
- API 構成
- DB
- ORM
- 認証
- アセット保存
- デプロイ先
- テスト方式
- 開発環境

を決定し、判断理由を記録する。

### フェーズ3: データモデル確定

- 学習内容カタログ
- 教材
- 問題
- 解答
- 解説
- アセット
- 目的別問題集
- 学習セッション
- 回答イベント
- 問題別進捗
- 定着状態

の初期 DB モデルとデータスキーマを確定する。

### フェーズ4: 基準問題セット

- 各問題形式の代表問題を作成する。
- DB へ登録する。
- スキーマ検証を行う。
- 画面・印刷・音声での扱いを確認する。

### フェーズ5: 完成版の縦方向実装

- 教材選択
- 出題
- 回答
- 判定
- 履歴保存
- 進捗計算
- 定着状態
- メディア表示
- 印刷

までを通す。

### フェーズ6: 本格教材作成

完成版のデータモデルと出題経路が安定した後、
新方式による正式教材の作成を開始する。

## 10. 新しいチャットへ切り替える条件

次が完了した時点で、
完成版再開発は新しいチャットへ切り替える。

- 移行計画書のコミット
- 保存用ブランチの作成
- 保存用タグの作成
- 保存用ブランチとタグの GitHub push
- `main` が完成版再構築を開始できる状態であることの確認
- 新チャット用の引き継ぎ文書の作成

新チャットでは、
現行試作の追加実装には戻らず、
完成版要求の統合から開始する。

## 11. 現時点の結論

現行試作は失敗ではなく、
完成版の要求を確認するための役割を完了した。

今後は現行コードや旧データを守るための作業を行わず、
実運用で得た要求と設計知識を使って、
完成版を最短経路で新規構築する。
