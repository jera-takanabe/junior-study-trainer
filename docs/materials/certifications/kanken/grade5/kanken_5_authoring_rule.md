# 漢検5級 questions.js 問題作成ルール v0.1

## 1. 目的

この文書は、タイムアタック一問一答アプリに投入する「漢検5級」問題データの作成ルールを定義する。

アプリ本体は、漢検・数学・理科・英検などに共通する汎用学習エンジンとして扱う。
本資料は、そのうち「漢検5級教材」を安定して増やすための教材作成ルールである。

---

## 2. 設計上の位置づけ

### 2.1 汎用設計との関係

- `quiz_progress_design_v0_3_generalized`  
  - アプリ全体の汎用設計
  - 問題別進捗、出題モード、習得判定、CSV出力などを定義する

- `questions_kanken5_authoring_rule_v0_1`  
  - 漢検5級の問題データ作成ルール
  - ID、カテゴリ、出題タイプ、targets、問題作成単位を定義する

### 2.2 設計メモを修正するか

現時点では、汎用設計メモの修正は必須ではない。

理由：

- 汎用設計メモは、漢検以外の数学・理科・英検にも使う上位設計である
- 今回決める内容は、漢検5級に特化した教材作成ルールである
- 上位設計に漢検固有ルールを入れすぎると、汎用性が下がる

ただし、将来的に以下のような共通ルールが固まった場合は、汎用設計メモへ反映する。

- `id` の命名規則を全カテゴリ共通化する
- `domain` の一覧を正式固定する
- `questionType` の一覧を全教材で共通管理する
- 習得判定ルールを questionType ごとに正式化する

## 3. 漢検5級問題データの基本構造

漢検5級問題は、以下の形式で作成する。

```javascript
{
  id: "kanken5-write-001",
  domain: "kanken",
  exam: "漢検5級",
  category: "漢検5級・書き練習",
  level: "書き取り",
  unit: "小6漢字",
  questionType: "write",
  question: "次のカタカナを漢字で書きなさい。\n\n計画をショウニンする。",
  answer: "承認",
  note: "承：うけたまわる。認：みとめる。",
  tags: ["漢検5級", "書き取り", "同音異字"],
  targets: ["承", "認", "承認"]
}
```

## 4. 必須項目

以下は必須とする。

```text
id
category
level
question
answer
```

ただし、漢検5級問題では、以下も原則として必ず入れる。

```text
domain
exam
unit
questionType
tags
targets
note
```

## 5. id 命名規則

### 5.1 基本形

```text
kanken5-{type}-{連番3桁}
```

例：

```text
kanken5-read-001
kanken5-write-001
kanken5-okuri-001
kanken5-error-001
kanken5-homophone-001
kanken5-idiom-001
```

### 5.2 type 一覧

```text
read       : 読み
write      : 書き取り
okuri      : 送り仮名
error      : 誤字訂正
meaning    : 意味
antonym    : 対義語
synonym    : 類義語
homophone  : 同音異字
idiom      : 四字熟語
structure  : 熟語の構成
radical    : 部首
stroke     : 筆順・画数
```

### 5.3 連番ルール

- 3桁連番にする
- 同じ type 内で重複させない
- 途中で削除しても番号は詰め直さない
- 問題文を大きく変更する場合は、新しいIDを作る

理由：

進捗データは `questionId` に紐づくため、IDを途中で変えたり再利用したりすると、過去の記録とズレる。

## 6. domain / exam

漢検5級では以下で固定する。

```javascript
domain: "kanken",
exam: "漢検5級"
```

---

## 7. category ルール

画面上の大分類として使う。

漢検5級では、まず以下の2分類で運用する。

```text
漢検5級・移動用
漢検5級・書き練習
```

### 7.1 漢検5級・移動用

スピンバイク、車移動、口頭確認に向く問題。

含めるもの：

```text
読み
意味選択
対義語
類義語
同音異字
四字熟語
熟語の構成
部首
画数
```

基本的に、息子本人が書かなくても答えられるもの。

### 7.2 漢検5級・書き練習

机で紙やノートに書く前提の問題。

含めるもの：

```text
書き取り
送り仮名
誤字訂正
漢字の使い分け
```

---

## 8. level ルール

`level` は、アプリ画面の「出題タイプ」として使う。

漢検5級では以下を使う。

```text
読み
選択・意味
選択・対義語
選択・類義語
選択・同音異字
選択・四字熟語
選択・熟語の構成
選択・部首
選択・画数
書き取り
送り仮名
誤字訂正
```

### 8.1 level の表記ルール

- 画面に表示されるので日本語でよい
- 表記ゆれを避ける
- 「意味選択」と「選択・意味」のような混在は禁止
- 選択問題は原則として `選択・〇〇` に統一する

---

## 9. questionType ルール

`questionType` は内部分類として使う。

漢検5級では以下を使う。

```text
read       : 読み
write      : 書き取り・送り仮名・誤字訂正
meaning    : 意味
antonym    : 対義語
synonym    : 類義語
homophone  : 同音異字
idiom      : 四字熟語
structure  : 熟語の構成
radical    : 部首
stroke     : 筆順・画数
```

### 9.1 level と questionType の対応

```text
読み                 -> read
書き取り             -> write
送り仮名             -> write
誤字訂正             -> write
選択・意味           -> meaning
選択・対義語         -> antonym
選択・類義語         -> synonym
選択・同音異字       -> homophone
選択・四字熟語       -> idiom
選択・熟語の構成     -> structure
選択・部首           -> radical
選択・画数           -> stroke
```

---

## 10. unit ルール

`unit` は教材上の範囲を表す。

漢検5級では、まず以下を使う。

```text
小6漢字
語句の意味
対義語
類義語
同音異字
四字熟語
熟語の構成
部首
筆順・画数
送り仮名
誤字訂正
```

厳密な単元というより、あとで絞り込み・分析しやすくする分類である。

---

## 11. tags ルール

`tags` は横断的な分類に使う。

### 11.1 基本タグ

必ず入れる：

```javascript
tags: ["漢検5級", "書き取り"]
```

または、

```javascript
tags: ["漢検5級", "読み"]
```

### 11.2 追加タグ例

```text
熟語
同音異字
送り仮名
対義語
類義語
四字熟語
誤字訂正
選択問題
口頭回答
机上練習
小6漢字
```

### 11.3 タグ運用ルール

- タグは増やしすぎない
- 同じ意味の表記ゆれを避ける
- 「書き」「書き取り」「書取」のような混在は禁止
- 迷った場合は既存タグに合わせる

---

## 12. targets ルール

`targets` は、その問題で身につけたい対象を表す。

### 12.1 漢字・熟語問題

漢字1字と熟語の両方を入れる。

例：

```javascript
targets: ["承", "認", "承認"]
```

理由：

- 1026字の進捗を見るには漢字1字が必要
- 実際の問題は熟語・文脈で出るため、熟語単位の記録も必要

### 12.2 送り仮名問題

```javascript
targets: ["険", "険しい"]
```

### 12.3 同音異字問題

正答だけでなく、混同しやすい語も入れる。

```javascript
targets: ["正確", "性格"]
```

### 12.4 四字熟語

```javascript
targets: ["有名無実"]
```

必要に応じて、構成漢字を入れてもよい。

```javascript
targets: ["有", "名", "無", "実", "有名無実"]
```

ただし、四字熟語の理解が目的の場合は、熟語単位を優先する。

## 13. 問題文作成ルール

### 13.1 読み問題

形式：

```text
次の漢字の読みは？

『承認』
```

答え：

```text
しょうにん
```

### 13.2 書き取り問題

形式：

```text
次のカタカナを漢字で書きなさい。

計画をショウニンする。
```

答え：

```text
承認
```

### 13.3 意味選択問題

形式：

```text
『貴重』の意味として正しいものは？

A. とても重い
B. とても大切で価値がある
C. とても速い
D. とても静か
```

答え：

```text
B. とても大切で価値がある
```

### 13.4 同音異字問題

形式：

```text
文に合う漢字はどれ？

先生の話をセイカクに聞く。

A. 性格
B. 正確
C. 政格
D. 成格
```

答え：

```text
B. 正確
```

### 13.5 誤字訂正問題

形式：

```text
まちがっている漢字を正しく直しなさい。

先生の説明を正格に聞く。
```

答え：

```text
正確
```

## 14. 移動用と書き練習用の分離

### 14.1 移動用に入れる問題

移動用は、口頭で答えられることを優先する。

入れてよい：

```text
読み
意味
対義語
類義語
同音異字の選択
四字熟語の意味
部首の選択
画数の選択
```

避ける：

```text
実際に書かないと力がつきにくい書き取り
細かい送り仮名
誤字訂正の記述式
```

### 14.2 書き練習用に入れる問題

書き練習用は、机上で手を動かすことを前提にする。

入れる：

```text
書き取り
送り仮名
誤字訂正
同音異字の書き分け
```

## 15. 1つの漢字に対して作る問題数

1026字すべてについて同じ問題数を作る必要はない。

### 15.1 基本方針

頻出・間違いやすい漢字は複数問作る。

例：

```text
承
  - 承認：読み
  - 承認：書き取り
  - 承知：意味
  - 継承：読み
```

一方で、簡単な漢字は1〜2問でもよい。

### 15.2 最小構成

まずは、1つの漢字につき最低1つは target に含まれる状態を目指す。

```text
1026字すべてが、少なくとも1問の targets に登場する
```

### 15.3 実用構成

最終的には以下を目指す。

```text
読み問題：1問
書き問題：1問
語句・用法問題：必要に応じて1問
```

すべての漢字に3問ずつ作る必要はない。

## 16. 習得判定との関係

アプリ側の初期ルール：

```text
read       : 2回連続正解で習得済み
write      : 3回連続正解で習得済み
choice系   : 3回連続正解で習得済み
```

漢検では、書けることが重要なので、書き問題は読み問題より厳しくする。

---

## 17. 問題追加の単位

いきなり1026字すべてを投入しない。

### 17.1 推奨する追加単位

```text
10問単位
20問単位
小さな漢字グループ単位
```

例：

```text
第1セット：承・認・誕・異・常・勤・勉・簡・単・貴
第2セット：険・従・温・結・果・賛・反・複・雑・確
```

### 17.2 追加後に確認すること

```text
カテゴリが正しく表示されるか
出題タイプが正しく表示されるか
ID重複がないか
未出題数が増えるか
進捗CSVに出るか
```

---

## 18. 問題作成時の品質ルール

### 18.1 避けること

```text
実際の過去問の丸写し
答えが複数あり得る曖昧な文
漢字のレベルが漢検5級から大きく外れる文
選択肢の正解が明らかすぎる問題
選択肢の日本語が不自然な問題
```

### 18.2 注意すること

```text
文脈で答えが一意に決まること
読みと書きを混同しないこと
送り仮名を答えに含める場合は note に明記すること
同音異字は混同しやすい語を選択肢に入れること
```

## 20. 今後の拡張候補

将来的には、以下を追加してもよい。

```text
source       : 出典や作成元
difficulty   : easy / normal / hard
priority     : high / normal / low
createdAt    : 作成日
updatedAt    : 更新日
```

ただし、現時点ではデータが重くなりすぎるため、必須にはしない。

## 21. 当面の運用方針

当面は、以下の順番で進める。

```text
1. このルールに沿って20〜30問作る
2. v0.3アプリで実際に使う
3. 進捗CSVを確認する
4. targets の粒度が適切か確認する
5. 問題追加ルールを微修正する
6. 50問、100問へ増やす
```

1026字を目指すが、最初から完璧な体系化を狙いすぎない。

まずは「増やしても壊れない形式」を守ることを優先する。
