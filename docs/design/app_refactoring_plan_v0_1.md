# アプリ分割・リファクタリング方針 v0.1

## 1. 目的

Junior Study Trainer の `app/index.html` は、HTML、CSS、JavaScript、出題制御、進捗管理、バックアップ処理などを1ファイルに含んでいる。
今後、目的別問題集、印刷用問題出力、図・表・グラフつき問題などを追加していくと、1つの変更が既存機能へ影響しやすくなる。

この文書では、既存アプリを壊さずに段階的にファイル分割・リファクタリングする方針を定義する。

## 2. 基本方針

- 既存アプリの動作を最優先で維持する。
- 一度に大きく分割しない。
- 新規機能から外部ファイル化する。
- 既存の出題・タイマー・進捗・履歴・バックアップ処理は、当面 `index.html` に残す。
- CSS は JavaScript より分離リスクが低いため、早い段階で `app/styles.css` に分離する。
- JavaScript は、目的別問題集などの新規機能から段階的に分離する。
- ES modules、ビルドツール、軽量フレームワークは当面導入しない。
- 将来的なサーバーサイド利用を見据え、特定フレームワークへの依存を急がない。

## 3. 当面の推奨構成

当面は、次のような構成を目指す。

    app/
    ├─ index.html
    ├─ styles.css
    ├─ question_sets_manifest.js
    ├─ purpose_sets_manifest.js
    ├─ purpose_sets.js
    ├─ questions.js
    └─ data/
       ├─ math_test1.js
       ├─ english_test1.js
       └─ ...

## 4. 分割順序

### 4.1 第1段階: CSS分離

`index.html` 内の style 要素を `app/styles.css` に移す。

目的:

- HTML構造と見た目を分離する。
- 今後のUI追加時にCSS差分を追いやすくする。
- JavaScriptを触らずに安全にファイル分割を開始する。

注意:

- class 名や id 名は変更しない。
- 見た目の改善は同時に行わない。
- まずは完全移植のみ行う。

### 4.2 第2段階: 目的別問題集補助JSの追加

`app/purpose_sets.js` を追加する。

役割:

- `window.PURPOSE_SETS_MANIFEST` の取得
- 目的別問題集一覧の基本整形
- draft / active などの状態判定
- UI表示に使う最小限の補助関数

注意:

- 既存の出題ロジックには接続しない。
- 既存の進捗・履歴には接続しない。
- 目的別問題集を選んでも、まだ出題開始できる状態にはしない。

### 4.3 第3段階: 目的別問題集UIの表示のみ追加

教材選択画面またはその周辺に、目的別問題集の一覧を表示する。

目的:

- 目的別問題集データがブラウザで読めていることを確認する。
- UI配置を確認する。
- 出題処理に接続する前に、表示だけで安全に確認する。

注意:

- クリックして出題開始する処理はまだ入れない。
- 進捗キーや履歴キーはまだ変更しない。
- 既存の教材選択UIを壊さない。

### 4.4 第4段階: 目的別問題集の出題接続

表示確認が済んだ後、目的別問題集を出題処理に接続する。

この段階では、次の設計を別途確認する。

- 進捗キーを `purposeSetId` 単位にするか
- 既存の `questionSetId` 単位進捗とどう共存させるか
- バックアップ対象に目的別問題集進捗を含めるか
- 目的別問題集の再開・リセットをどう扱うか

## 5. 当面分割しないもの

以下は、依存関係が強いため当面分割しない。

- 出題中の状態管理
- タイマー処理
- 正誤記録
- 進捗保存
- 履歴保存
- バックアップと復元
- 既存教材の読み込み制御

これらは目的別問題集の表示・確認が終わった後、必要に応じて段階的に分離する。

## 6. 将来の分割候補

将来的には次の分割を検討する。

    app/
    ├─ storage.js
    ├─ quiz_engine.js
    ├─ timer.js
    ├─ question_loader.js
    ├─ ui_helpers.js
    └─ print_export.js

ただし、これらはすぐに実施しない。
既存動作を十分に確認しながら、1ファイルずつ分離する。

## 7. CSS分割方針

CSSは `app/styles.css` に分離する。

方針:

- まずは既存CSSの完全移植のみ。
- セレクタ名は変更しない。
- デザイン調整は同時に行わない。
- 目的別問題集UI用CSSは、後続作業で追記する。

HTML側では、`style` 要素を削除し、次を追加する。

    <link rel="stylesheet" href="styles.css">

## 8. JavaScript分割方針

JavaScriptは、既存機能ではなく新規機能から分離する。

当面の追加候補:

- `app/purpose_sets.js`

既存の大きな処理は、目的別問題集UIが安定するまで `index.html` に残す。

## 9. グローバル名前空間の方針

当面は ES modules を使わず、通常の script 読み込みを継続する。

新規JSは、グローバル汚染を避けるため、必要最小限の名前だけを `window` に公開する。

例:

    window.PurposeSets = {
      getAll() {
        return Array.isArray(window.PURPOSE_SETS_MANIFEST)
          ? window.PURPOSE_SETS_MANIFEST
          : [];
      }
    };

## 10. フレームワーク方針

当面、軽量フレームワークは導入しない。

理由:

- 現在のアプリは GitHub Pages 上で静的に動作している。
- 既存構成はビルド不要で、ユーザーが Git Bash で確認しやすい。
- 目的別問題集は、まだデータ定義と表示確認の段階である。
- 将来的なサーバーサイド利用を考えると、現時点で特定フレームワークへ寄せすぎない方がよい。

再検討タイミング:

- 教材選択UIが複雑化したとき
- 目的別問題集の検索・絞り込み・タブ切替が増えたとき
- 進捗ダッシュボードを本格実装するとき
- 印刷用画面をアプリ内に統合するとき
- DOM操作が保守困難になったとき

## 11. サーバーサイド利用を見据えた注意

将来サーバーサイドを利用する可能性があるため、次を意識する。

- データ構造をUIフレームワークに強く依存させない。
- `questionSetId`, `questionId`, `purposeSetId` などのID体系を安定させる。
- ローカル保存前提の処理と、将来サーバー保存に移せる処理を分けて考える。
- ビジネスロジックに近い処理は、将来的に `quiz_engine.js` や `storage.js` へ分離できるようにする。

## 12. この文書作成時点の判断

目的別問題集UIに入る前に、まず次の順で進める。

1. このリファクタリング方針を文書化する。
2. CSSを `app/styles.css` に分離する。
3. `app/purpose_sets.js` を追加する。
4. 目的別問題集UIを表示のみで追加する。
5. 出題処理への接続は、進捗・履歴設計を確認してから行う。
