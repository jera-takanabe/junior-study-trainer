# 目的別問題集 実行時設計 v0.1

## 1. 目的

この文書は、Junior Study Trainer における目的別問題集を、既存の通常教材と安全に共存させながら実行するための設計方針を定義する。

目的別問題集は、通常教材の中から特定の問題を明示IDで集めた学習単位である。
そのため、通常教材の `questionSetId` 単位の進捗・履歴と、目的別問題集の `purposeSetId` 単位の進捗・履歴を混同しないことが重要である。

## 2. 現在の前提

現時点では、次の状態まで実装済みである。

- `app/purpose_sets_manifest.js` に目的別問題集の定義がある。
- `app/purpose_sets.js` に目的別問題集の取得・表示補助がある。
- `app/index.html` で目的別問題集パネルを表示している。
- `status: active` の目的別問題集は、現在読み込み済み教材と同じ `questionSetId` を参照している場合だけ開始できる。
- 目的別問題集は、まず保存なし実行として出題に接続済みである。
- 保存なし実行では、通常教材の進捗・履歴へ記録しない。
- 結果画面には、目的別問題集名と保存なし実行であることを表示する。
- 目的別問題集の「同じ設定でもう一度」「間違えた問題だけ再挑戦」も保存なしを維持する。
- 目的別問題集Manifest側の `limit`, `timer`, `ordering`, `randomization` を実行時に反映する。
- 目的別問題集パネルに、出題数・問題表示秒数・回答表示秒数・出題順を表示する。
- 複数の目的別問題集を同時に表示・実行できる。
- 数学だけでなく、英語教材の目的別問題集も追加済みである。
- 現在教材と一致する目的別問題集だけ開始可能になり、他教材の目的別問題集は開始不可になることを確認済みである。
- 既存の通常教材出題、進捗、履歴、バックアップは従来通り動作している。

## 3. 通常教材と目的別問題集の違い

### 3.1 通常教材

通常教材は、1つの `questionSetId` に紐づく問題データ全体を出題対象とする。

例:

    questionSetId: school_math_jhs1_test1

進捗・履歴は、現在この `questionSetId` 単位で保存されている。

### 3.2 目的別問題集

目的別問題集は、1つ以上の通常教材から、特定の問題を選んだ学習単位である。

例:

    purposeSetId: purpose_math_test1_basic_check_001

目的別問題集内の各問題は、次の組で参照する。

    questionSetId + questionId

この組により、同じ `questionId` が将来別教材に存在しても参照先を一意にできる。

## 3.5 学習管理レイヤー

目的別問題集の本格運用では、学習管理を次の2系統に分ける。

    周回管理:
      短期対策用。
      前回間違えた問題だけを次周へ残し、間違いがなくなったら完了とする。

    弱点管理:
      長期育成用。
      誤答回数、誤答率、前回実施からの経過日数、正解までの時間などをもとに、忘れやすい問題を管理する。

周回管理と弱点管理は、似ているが目的が異なる。

周回管理は、定期テスト、単元テスト、漢字テスト、英単語小テストなど、範囲が限定される短期対策に向いている。
弱点管理は、全国模試、英検、数検、長期復習など、範囲が広く、忘却を拾う必要がある学習に向いている。

両者は同じ問題を対象にしてもよいが、状態とリセット単位は分ける。

    周回管理のリセット:
      purposeSetId 単位またはテスト対策単位で行う。

    弱点管理のリセット:
      長期弱点データとして別に行う。

    通常履歴のリセット:
      通常教材または目的別問題集の履歴リセットとして別に扱う。

周回で正解したことは、弱点解除と同義ではない。
テスト直前に正解できても、一定期間後に覚えているとは限らないためである。

連動方針は次のようにする。

- 周回で間違えた問題は、弱点スコアにも反映してよい。
- 周回で正解した問題は、弱点スコアを少し下げてもよい。
- 周回完了は、弱点解除とは限らない。
- 弱点解除には、一定期間を空けた正解を条件にする。

弱点管理の初期アルゴリズム案:

    wrongPoint: 2
    passPoint: 1
    correctPoint: -1
    weaknessThreshold: 3
    releaseThreshold: 0
    retentionDaysForRelease: 7
    recentWindowSize: 5

初期実装では、このアルゴリズムをすぐ実装しなくてもよい。
ただし、保存設計では、将来 `weaknessScore` や `weaknessStatus` を持てる余地を残す。

## 3.6 出題モード

目的別問題集の本格運用では、出題対象の決め方を「出題モード」として分ける。

    通常練習:
      条件に合う問題を通常どおり出題する。

    周回練習:
      前回間違えた問題、または前回パスした問題だけを出題する。
      対象がなくなったら周回完了とする。

    弱点練習:
      weaknessScore が高い問題、または weaknessStatus が弱点の問題を出題する。

    定着確認:
      一定期間空いた弱点問題を再確認する。
      7日以上、14日以上などの間隔を設定可能にする。

    ランダム確認:
      対象範囲からランダムに出題する。

定期テスト向けの目的別問題集では、まず周回練習を優先する。
全国模試、検定、長期復習では、弱点練習と定着確認を重視する。

出題モードは、目的別問題集そのものとは分けて扱う。

例:

    数学 第1回定期テスト 文章題確認 × 周回練習
    英語 Get Ready 基本確認 × 周回練習
    英検3級単語 × 弱点練習
    数検6級復習 × 定着確認
    全国模試対策 × 弱点練習

この分離により、同じ目的別問題集でも、通常練習、周回練習、弱点練習を切り替えられる。

## 4. 実行時の基本方針

目的別問題集を実行する場合、通常教材とは別の実行モードとして扱う。

    通常教材モード:
      questionSetId を実行単位にする

    目的別問題集モード:
      purposeSetId を実行単位にする

目的別問題集モードでは、内部的には通常教材の問題データを参照するが、進捗・履歴・再開・リセットの単位は `purposeSetId` とする。

## 5. 進捗キー方針

### 5.1 通常教材の進捗キー

通常教材の進捗キーは既存のまま維持する。

例:

    timedQuizTrainerProgressV04__school_math_jhs1_test1

### 5.2 目的別問題集の進捗キー

目的別問題集の進捗は、通常教材と混ぜない。

例:

    timedQuizTrainerPurposeProgressV01__purpose_math_test1_basic_check_001

理由:

- 通常教材での習得状況と、目的別問題集での習得状況は目的が異なる。
- 目的別問題集は短期対策・弱点補強・直前確認などの用途が想定される。
- 同じ問題でも、通常教材での進捗と目的別問題集での進捗を別々に持てる方が安全である。

## 6. 履歴キー方針

### 6.1 通常教材の履歴キー

通常教材の履歴キーは既存のまま維持する。

例:

    timedQuizTrainerHistoryV04__school_math_jhs1_test1

### 6.2 目的別問題集の履歴キー

目的別問題集の履歴は、通常教材と別キーにする。

例:

    timedQuizTrainerPurposeHistoryV01__purpose_math_test1_basic_check_001

履歴には、少なくとも次を保存する。

- 実行モード
- `purposeSetId`
- 目的別問題集名
- 参照元 `questionSetId`
- `questionId`
- 正誤結果
- 実行日時
- 問題表示秒数
- 回答表示秒数

## 7. 問題データ解決方針

目的別問題集は、`purposeSetId` から次の手順で出題用問題配列を作る。

1. `purposeSetId` から目的別問題集定義を取得する。
2. `questions[]` の各参照を読む。
3. 参照先の `questionSetId` ごとに必要な教材データを読み込む。
4. `questionId` で問題を取得する。
5. 目的別問題集の順序に従って出題配列を作る。

当面は、1つの目的別問題集が1つの通常教材だけを参照する前提で実装してよい。
ただし、設計上は複数教材参照を禁止しない。

## 8. 初期実装の制限

初期実装では、次の制限を置く。

- 目的別問題集は `selectionMode: explicitIds` のみ対応する。
- 参照先教材は、最初は現在読み込み済み教材と同じ `questionSetId` のみ対応する。
- 複数教材をまたぐ目的別問題集は、設計上は許容するが実装は後回しにする。
- 目的別問題集の進捗は、通常教材へ反映しない。
- 通常教材の進捗は、目的別問題集へ反映しない。
- バックアップ対応は、出題接続後に別途実施する。

## 9. UI方針

目的別問題集パネルには、当面次を表示する。

- 目的別問題集名
- `purposeSetId`
- 問題数
- 状態
- 説明
- 開始可否
- 保存なし実行であること

出題接続時には、通常教材一覧の「この教材を選択」と混同しないように、目的別問題集側のボタン名は別にする。

候補:

    この目的別問題集を始める
    目的別問題集で練習する
    下書きのため開始不可

`status: draft` の目的別問題集は開始不可にする。
`status: active` で、かつ現在読み込み済み教材と同じ `questionSetId` だけで解決できる場合に開始可能にする。

初期実装の開始ボタンは次の文言を使う。

    この目的別問題集を始める

開始可能な目的別問題集には、保存なし実行であることを明示する。

## 10. 実行モードの追加

アプリ内部には、目的別問題集の保存なし実行を区別するため、`runtimeContext` を持たせる。

現在の実装:

    runtimeContext:
      mode: "questionSet" | "purposeSet"
      id: questionSetId | purposeSetId
      title: 表示名
      saveProgress: true | false
      saveHistory: true | false

目的別問題集の実行時設定:

    limit:
      出題数
    timer.questionSeconds:
      問題表示秒数
    timer.answerSeconds:
      回答表示秒数
    ordering:
      fixed または random
    randomization:
      true の場合はランダム出題

通常教材の場合:

    mode = "questionSet"
    id = questionSetId
    saveProgress = true
    saveHistory = true

目的別問題集の初期実装の場合:

    mode = "purposeSet"
    id = purposeSetId
    saveProgress = false
    saveHistory = false

この設計により、出題処理、履歴、進捗、バックアップの分岐を明確にできる。

## 11. バックアップ方針

初期の出題接続では、バックアップ対応を同時に行わない。

理由:

- 現在のバックアップは通常教材単位を前提としている。
- 目的別問題集の進捗・履歴を混ぜると復元時の事故が起きやすい。
- まずは目的別問題集の出題と進捗保存を安定させる。

将来的には、バックアップJSONに次を含めることを検討する。

- 通常教材の履歴・進捗
- 目的別問題集の履歴・進捗
- 実行モード
- 対象ID
- バックアップ形式バージョン

## 12. リセット方針

目的別問題集の進捗リセットは、通常教材の進捗リセットとは分ける。

通常教材:

    questionSetId 単位でリセット

目的別問題集:

    purposeSetId 単位でリセット

初期実装では、目的別問題集の進捗リセットUIは後回しにしてよい。

## 13. 実装順序

目的別問題集の出題接続は、次の順で進める。

1. この実行時設計を文書化する。
2. `app/purpose_sets.js` に、目的別問題集から出題参照を取得する補助関数を追加する。
3. 現在読み込み済み教材と同じ `questionSetId` の目的別問題集だけ、出題候補として解決できるようにする。
4. 目的別問題集パネルに開始ボタンを表示する。
5. 開始ボタン押下時に、通常教材とは別の設定で出題配列を作る。
6. まずは履歴・進捗保存なし、または一時的なメモリ実行で確認する。
7. 結果画面に目的別問題集名と保存なし実行であることを表示する。
8. 「同じ設定でもう一度」「間違えた問題だけ再挑戦」でも保存なしを維持する。
9. 問題なく動作した後、`purposeSetId` 単位の進捗・履歴保存を追加する。
10. 最後にバックアップ・リセット・CSV出力への対応を検討する。

## 14. 当面の判断

現在の実装では、まだ完全な目的別問題集運用には進まない。

実装済みの補助関数:

- `getQuestionRefs(purposeSetId)`
- `canResolveWithCurrentQuestionSet(purposeSetId, currentQuestionSetId)`
- `resolveQuestionsFromCurrentSet(purposeSetId, currentQuestionSetId, questions)`

これにより、目的別問題集が現在読み込み済み教材だけで出題可能かを判定できる。

実装済みの実行範囲:

- 現在読み込み済み教材と同じ `questionSetId` の目的別問題集だけ開始できる。
- `selectionMode: explicitIds` の目的別問題集だけ対応する。
- 目的別問題集の `limit` または参照問題数どおりに出題する。
- 目的別問題集では、設定画面の問題数指定による切り詰めを行わない。
- 目的別問題集Manifest側の `timer` を問題表示秒数・回答表示秒数に反映する。
- `randomization: true` または `ordering: "random"` の場合だけランダム出題にする。
- `ordering: "fixed"` かつ `randomization: false` の場合は固定順にする。
- 複数の目的別問題集を同時に表示・実行できる。
- 数学教材と英語教材の目的別問題集を同時に登録できる。
- 現在読み込み中の教材と一致しない目的別問題集は開始不可として表示できる。
- 現在教材で開始可能な目的別問題集を上に表示できる。
- 目的別問題集パネルに、出題数・問題表示秒数・回答表示秒数・出題順を表示できる。
- 結果画面まで表示できる。
- 結果画面に目的別問題集名と保存なし実行であることを表示する。
- 復習欄のセット表示には目的別問題集名を表示する。
- 「同じ設定でもう一度」「間違えた問題だけ再挑戦」でも保存なしを維持する。
