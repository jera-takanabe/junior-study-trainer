# メタデータ移行計画 v0.1

## 1. この文書の目的

この文書は、Junior Study Trainer において、既存の問題データを壊さずに共通メタデータ設計へ段階的に移行するための計画を整理する設計メモである。

目的は、すぐに `app/data/*.js` を大規模変更することではなく、今後の新規教材・目的別問題集・印刷出力・図表つき問題・音声問題へ進むための安全な移行手順を決めることである。

関連する設計文書は次の通り。

```text
関連設計
├─ catalog_to_question_data_design_v0_1.md
├─ purpose_based_question_sets_design_v0_1.md
├─ printable_question_output_design_v0_1.md
├─ visual_question_data_design_v0_1.md
├─ gui_structure_design_v0_1.md
└─ question_metadata_schema_design_v0_1.md
```

## 2. 現在の前提

現在のアプリは、既存の問題データ形式で安定して動作している。

```text
現在できていること
├─ 教材選択
├─ 動的読み込み
├─ タイマー付き一問一答
├─ 手動正誤
├─ 教材単位の履歴
├─ 教材単位の進捗
└─ 選択教材単位のバックアップ
```

この安定性を優先し、既存形式を一気に置き換えない。

## 3. 基本方針

```text
基本方針
├─ 既存 app/data/*.js は当面そのまま動かす
├─ 既存問題への一括メタデータ付与は行わない
├─ 新規教材から metadata を任意追加する
├─ アプリ側は未知の metadata を無視できるようにする
├─ 目的別問題集は最初は明示ID方式から試す
├─ 抽出条件方式はタグが十分そろってから試す
├─ 印刷出力はアプリ外スクリプトから始める
├─ 図表つき問題は自作画像1枚のプロトタイプから始める
├─ 音声・スピーキングは後期に回す
└─ すべてを一度に実装しない
```

## 4. 移行しないものを明確にする

当面、次は移行しない。

```text
当面移行しないもの
├─ 既存全問題への catalogRefs 付与
├─ 既存全問題への domainTags 付与
├─ 既存全問題への print 属性付与
├─ 既存全問題への visual 属性付与
├─ 既存全問題への audio 属性付与
├─ 既存全問題への speaking 属性付与
├─ 進捗データの内部構造変更
└─ バックアップ形式の大幅変更
```

理由：

```text
理由
├─ 既存アプリの安定運用を優先する
├─ 既存問題数が多く、一括タグ付けの負荷が高い
├─ タグ設計は実際の教材作成で調整が入る
├─ 目的別問題集の実装前に全移行しても効果が薄い
└─ 手戻り時の修正範囲を小さくする
```

## 5. 新規問題データでの導入方針

新規教材・新規問題では、可能な範囲で `metadata` を追加する。

```text
新規問題で優先する metadata
├─ id
├─ questionSetId
├─ subjectType
├─ school または certification
├─ domainTags
├─ questionTypeTags
├─ knowledgeTags
├─ skillTags
├─ purposeTags
├─ source
├─ print
└─ status
```

ただし、すべてを必須にはしない。

```text
最初の必須候補
├─ id
├─ questionSetId
├─ domainTags
├─ questionTypeTags
└─ status
```

## 6. 既存問題へのタグ付け優先順位

既存問題へタグを追加する場合は、次の順を推奨する。

```text
優先順位1：
  直近で使う検定対策
  ├─ 数検6級
  ├─ 数検5級
  ├─ 英検3級
  └─ 漢検4級

優先順位2：
  次回定期テスト対策

優先順位3：
  スピンバイク・車移動用の短時間問題

優先順位4：
  印刷用にしたい問題

優先順位5：
  図表・音声などの特殊問題
```

既存の全教材に一律でタグ付けするのではなく、利用予定が近いものから行う。

## 7. 最初に試す目的別問題集

目的別問題集は、まず明示ID方式で試す。

```text
最初に試す候補
├─ 数検6級 薄い総点検
├─ 数検5級 正負の数
├─ 英検3級 語彙
├─ 漢検4級 読み
└─ スピンバイク40問
```

推奨は、数検6級またはスピンバイク40問である。

理由：

```text
数検6級：
  範囲確認が目的で、問題数を少なくできる

スピンバイク40問：
  既存のアプリ利用目的と相性がよい

英検3級語彙：
  音声なしでも始められる

漢検4級読み：
  書取よりアプリ向き
```

## 8. 抽出条件方式への移行条件

抽出条件方式は、次がそろってから導入する。

```text
導入条件
├─ 問題に domainTags がある
├─ 問題に questionTypeTags がある
├─ 問題に purposeTags がある
├─ questionSetId をまたいだ抽出が必要になっている
├─ 明示ID方式では管理が重くなっている
└─ GUI上で目的から選ぶ導線が必要になっている
```

それまでは、明示ID方式で十分である。

## 9. 印刷出力の移行計画

印刷出力は、アプリ本体ではなく、まずローカルスクリプトまたは手動Markdown生成で試す。

```text
段階1：
  ChatGPTで問題用紙Markdownと解答Markdownを作る

段階2：
  app/data/*.js から手動で問題を抜き出す

段階3：
  ローカルスクリプトでMarkdownを生成する

段階4：
  PDF化する

段階5：
  アプリUIに印刷ボタンを追加する
```

最初に向いている印刷対象：

```text
印刷向き初期候補
├─ 漢検4級 書取
├─ 数検5級 方程式文章題
├─ 英検3級 Writing
├─ 数学定期テスト 文章題
└─ 作図なしの総合問題
```

## 10. 図表つき問題の移行計画

図表つき問題は、まず自作画像1枚から試す。

```text
段階1：
  自作画像を1枚作る

段階2：
  app/assets/visuals/ に置く

段階3：
  問題データから参照する

段階4：
  画面に表示する

段階5：
  印刷Markdownにも出す

段階6：
  表・グラフの構造化へ進む
```

初期候補：

```text
図表つき初期候補
├─ 数学の簡単な図形
├─ 数検5級の座標平面
├─ 理科の模式図
├─ 社会の自作白地図
└─ 英検3級面接カード風イラスト
```

## 11. 音声・スピーキングの移行計画

音声・スピーキングは後期に回す。

```text
音声再生
├─ 英検3級リスニング
├─ 英単語音声
├─ 音読練習
└─ シャドーイング

スピーキング
├─ 英検3級二次試験
├─ 音読
├─ 質問応答
├─ イラスト描写
└─ AI評価
```

導入条件：

```text
導入条件
├─ 問題メタデータに audio / speaking がある
├─ 音声ファイルの配置方針が決まっている
├─ 著作権上使える音声がある
├─ GUI上に Listening / Speaking の入口がある
└─ まずは評価なしの録音・自己確認から始められる
```

## 12. 進捗データの移行計画

進捗データは慎重に扱う。

```text
現行：
  questionSetId 単位

次段階：
  purposeSetId 単位を追加

将来：
  questionId 単位
  catalogRefs 単位
  tag 単位
```

当面の方針：

```text
当面
├─ 既存進捗形式は維持
├─ purposeSetId の進捗は別キーで追加
├─ 既存バックアップ形式を壊さない
├─ 旧形式から新形式への自動変換は後回し
└─ リセット単位を分かりやすくする
```

## 13. バックアップ形式への影響

メタデータ導入後も、バックアップは壊さない。

```text
バックアップ方針
├─ 既存バックアップを読めるようにする
├─ 新しい進捗キーを追加しても旧キーを残す
├─ 不明な項目は無視できるようにする
├─ バージョン番号を持たせる
└─ 復元時にエラーで止まらないようにする
```

将来的には、バックアップ対象を次のように拡張する。

```text
将来のバックアップ対象
├─ 教材別進捗
├─ 目的別問題集進捗
├─ 問題単位履歴
├─ 弱点情報
├─ 設定
└─ 印刷履歴
```

## 15. 最初の実装候補

最初に実装へ移すなら、次のどれかがよい。

```text
候補A：
  目的別問題集の明示ID方式

候補B：
  印刷用Markdown生成

候補C：
  新規問題への metadata 付与ルール

候補D：
  図表画像1枚表示プロトタイプ
```

推奨は候補Cと候補Aの組み合わせである。

理由：

```text
理由
├─ 既存アプリを壊さない
├─ すぐに新規教材作成へ活かせる
├─ 目的別問題集の効果を確認できる
├─ 抽出条件方式の前段階になる
└─ 実装量が比較的小さい
```

## 17. 既存教材への暫定対応

既存教材は、次のように扱う。

```text
既存教材
├─ そのまま利用する
├─ 必要な問題だけIDを確認する
├─ 目的別問題集で使う問題だけタグを補う
├─ 印刷に使う問題だけ print を補う
└─ 図表が必要な問題だけ visual を補う
```

全件移行はしない。

## 18. リスクと対策

### 18.1 タグが増えすぎる

```text
リスク：
  タグが増えすぎて管理できなくなる

対策：
  domainTags / questionTypeTags / knowledgeTags / skillTags を分ける
  最初は少数タグで始める
  カタログを基準にする
```

### 18.2 GUIが複雑になる

```text
リスク：
  子どもが迷う

対策：
  子ども用は「今日やる」「前回の続き」「スピンバイク」程度に絞る
  詳細設定は親用に寄せる
```

### 18.3 既存進捗が壊れる

```text
リスク：
  進捗・履歴が失われる

対策：
  進捗形式は当面変更しない
  新キーは追加方式にする
  バックアップ互換性を維持する
```

### 18.4 著作権資料が混入する

```text
リスク：
  原本画像や本文が公開リポジトリに入る

対策：
  .local/ 運用を徹底する
  source.localOnly を使う
  repositoryAllowed を明示する
```

## 21. まとめ

メタデータ移行は、大工事ではなく段階導入とする。

```text
今すぐ：
  既存アプリを壊さない
  新規問題から metadata を意識する
  明示ID方式の目的別問題集を試す

次：
  印刷Markdown生成
  図表画像1枚表示
  タグ抽出

将来：
  目的別進捗
  弱点自動抽出
  音声
  スピーキングAI評価
```

この方針により、既存アプリを安定運用しながら、Junior Study Trainer を段階的に拡張できる。
