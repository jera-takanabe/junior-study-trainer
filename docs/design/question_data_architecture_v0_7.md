# Question Data Architecture v0.7

## 1. この設計書の目的

この設計書は、Junior Study Trainer における問題データ構造と配置方針を、v0.7 時点の前提として整理するためのものである。

現在のアプリは、漢検5級 Set001〜Set010 のようなテキスト中心の一問一答には対応できている。一方で、今後追加したい学校教科の問題では、図、表、グラフ、数式、長文、会話文、地図、年表などを扱う可能性がある。

そのため、学校教科の問題作成に進む前に、現行の `app/questions.js` 1本運用をどのように位置づけるか、問題データをどの単位で分割するか、実行時にどこから読み込むか、将来的に `data/` 配下の教材データと `app/` 配下の実行用データをどう関係づけるかを整理する。

v0.7 では、すぐに大規模な実装変更を行うのではなく、次の学校教科プロトタイプに進むための暫定方針を決めることを目的とする。

## 2. 現在の状態

### 2.1 現行の問題データ配置

現在の問題データは、大きく分けて次の2種類で管理している。

- `data/certifications/kanken/grade5/sets/` 配下のセット別データ
- `app/questions.js` の実行用統合データ

漢検5級については、Set001〜Set010 の問題データがセット別に作成済みである。また、アプリ実行時には `app/questions.js` を読み込むことで、Set001〜Set010 の合計438問を表示できる状態になっている。

### 2.2 現行の GitHub Pages 公開方式

現在の GitHub Pages 公開方式では、`app/` フォルダのみを公開対象としている。

そのため、公開アプリが直接参照できるファイルは、基本的に `app/` 配下に配置されたファイルである。現在は、`app/index.html` と `app/questions.js` を中心に公開されている。

### 2.3 現行方式のメリット

現行方式は、構成が単純である。

アプリ側は `app/questions.js` を読み込めばよく、複数ファイルの読み込み制御やビルド処理を必要としない。そのため、漢検5級のようなテキスト中心の一問一答を早く動かすには適している。

また、GitHub Pages への公開対象が `app/` に限定されているため、公開範囲も分かりやすい。

### 2.4 現行方式の限界

一方で、すべての問題を `app/questions.js` 1本に統合する方式には限界がある。

問題数が増えると、ファイルが大きくなり、教材単位・カテゴリ単位・セット単位での管理が難しくなる。また、学校教科のように、図、表、グラフ、数式、長文、会話文、地図、年表などを扱う問題では、単純な `question` 文字列中心の構造だけでは表現しにくい可能性がある。

そのため、v0.7 では、現行方式をすぐに廃止するのではなく、現行方式の位置づけを明確にしたうえで、次段階のデータ分割と拡張問題形式を検討する。

## 3. v0.7 で整理したい論点

v0.7 では、最終的なWebアプリ化・DB管理を前提にしつつ、そこへ一気に進むのではなく、現在の静的アプリ運用から次の段階へ進むための過渡期の方針を整理する。

この設計書で扱う問題データ分割は、最終形ではない。あくまでも、DB化する前の段階で、教材データを安全に増やし、メンテナンスしやすくするための中間設計である。

### 3.1 最終的に目指す姿

最終的には、問題データをWebアプリからDBで管理する構成を目指す。

DB管理に移行すれば、問題の追加、修正、分類、検索、出題履歴、正答率、ユーザー別進捗などを、ファイル単位ではなくデータとして扱えるようになる。

ただし、現時点で一気にWebアプリ化・DB化を行うと、アプリ本体、データモデル、管理画面、サーバー構成、デプロイ、運用の検討範囲が一気に広がる。

そのため、現在は小さく動く構成を維持しながら、段階的に移行する。

### 3.2 現在の1ファイル運用の位置づけ

現在の `app/questions.js` 1ファイル運用は、初期プロトタイプとして妥当な方式である。

特に、漢検5級 Set001〜Set010 までは、まず動く教材を作ることを優先し、問題データの分割よりも、アプリで問題を出せる状態にすることを優先した。

この方式は最終形ではないが、プロトタイプ段階では、構成が単純で壊れにくいというメリットがある。

### 3.3 Set010以降で分割を検討する理由

Set001〜Set010 まで到達したことで、問題数は438問になった。

ここから先も `app/questions.js` 1本に追加し続けると、ファイルが大きくなり、セット単位での確認、修正、差し替えが難しくなる。

そのため、Set010以降では、DB化に進む前の過渡期として、問題データの分割方式を検討する。

ただし、この分割は「最終的な理想構造」を作るためではなく、現在の静的アプリ運用をもう少し安全に続けるための整理である。

### 3.4 過渡期として決めること

v0.7 で決めるべきことは、最終的なDB設計ではなく、次の過渡期の運用方針である。

主に次の点を整理する。

- `app/questions.js` 1本運用を当面維持するか
- 分割する場合、`app/data/...` に実行用データを置くか
- `data/...` 配下の教材データをどの程度「原本」として扱うか
- 分割ファイルを手動で管理するか、DB化までの一時的な生成方式を設けるか
- 学校教科プロトタイプでは、どの程度まで拡張問題形式を試すか

### 3.5 v0.7 でまだ決めないこと

v0.7 では、最終的なDB設計までは決めない。

具体的には、次の内容は今後の検討対象とする。

- DBの種類
- サーバー構成
- 認証方式
- ユーザー別進捗管理
- 管理画面
- 本番運用方式
- AWSなどのクラウド配置方式

これらは重要だが、今この段階で詳細化すると設計範囲が広がりすぎる。

そのため、v0.7 では、最終的にDB化する前提を明記したうえで、現在の静的アプリから次の一歩として何を分割し、何をまだ分割しないかを整理する。

## 4. 過渡期の問題データ配置方針

v0.7 では、最終的なDB管理に進む前の過渡期として、問題データの配置方針を整理する。

この章で扱う配置方針は、最終的なWebアプリ・DB構成ではない。現在の静的アプリ運用を維持しながら、問題数や教材カテゴリが増えても管理しやすくするための中間的な方針である。

### 4.1 現行方式: app/questions.js 1本

現行方式では、アプリ実行時に `app/questions.js` を読み込む。

この方式は、初期プロトタイプとしては分かりやすく、GitHub Pagesでも扱いやすい。`app/index.html` と `app/questions.js` があれば動作するため、構成が単純である。

漢検5級 Set001〜Set010 までは、この方式で進めることにより、まず動く教材を作ることを優先した。

一方で、今後さらに問題数や教材カテゴリが増える場合、`app/questions.js` 1本にすべてを追加し続けると、ファイルが肥大化し、確認・修正・差し替えが難しくなる。

### 4.2 過渡期方式: app/data/... に実行用データを分割配置

Set010以降では、過渡期の方式として、`app/data/...` に実行用データを分割配置する案を検討する。

この方式では、GitHub Pagesで公開される `app/` 配下に、アプリが直接読み込める問題データを置く。

例:

```text
app/
  index.html
  questions.js
  data/
    certifications/
      kanken/
        grade5/
          s001.js
          s002.js
          ...
    school/
      math/
        jh1/
          positive_negative_numbers/
            s001.js
```

この方式の目的は、最終的なDB構成を先取りすることではない。

あくまでも、静的アプリのまま、セット単位・教材単位でファイルを分け、修正や追加をしやすくするための過渡期の整理である。

### 4.3 data/... 配下の位置づけ

`data/...` 配下は、教材データの作成・保管場所として扱う。

ただし、現在のGitHub Pages公開方式では `app/` のみを公開対象としているため、`data/...` 配下のファイルは、そのままでは公開アプリから直接読み込めない。

そのため、v0.7時点では、`data/...` を「作者・管理用の教材データ置き場」として扱い、実行時に読み込むデータは `app/` 配下に置く方針を基本とする。

将来的には、`data/...` から `app/data/...` へコピーまたは生成する方式を検討できる。ただし、これはDB化までの一時的な運用であり、最終形ではない。

### 4.4 最終形: Webアプリ + DB管理

最終的には、問題データをファイルではなくDBで管理する構成を目指す。

DB管理に移行すれば、問題の追加、修正、分類、検索、出題履歴、正答率、ユーザー別進捗などを、アプリの機能として扱いやすくなる。

そのため、v0.7のファイル分割は、DB化に置き換わる前提の過渡期設計である。

この段階で過度に複雑なファイル構造を作り込むのではなく、次の学校教科プロトタイプを安全に進められる程度の分割に留める。

## 5. 過渡期の問題データ表現方針

v0.7 では、最終的なDB設計ではなく、静的ファイル運用のまま学校教科プロトタイプを進めるための問題データ表現を整理する。

ここで定義する問題データ表現は、DB化までの過渡期のためのものである。ただし、将来DB化するときに大きく作り直さなくて済むように、問題本文、補助資料、解答、解説、分類情報をできるだけ分けて考える。

### 5.1 テキスト中心問題

漢検5級のような一問一答では、テキスト中心の問題形式で対応できる。

基本的には、次の情報を持てればよい。

- 問題ID
- セットID
- 問題文
- 解答
- 解説
- カテゴリ
- タグ

この形式は、漢字、語句、英単語、短い計算問題などに向いている。

### 5.2 画像・図表付き問題

学校教科では、問題文だけでなく、画像、図、表、グラフなどを見て答える問題が必要になる可能性がある。

この場合、問題文の中に画像や図表を直接埋め込むのではなく、問題本文と補助資料を分けて持つ方がよい。

過渡期の静的ファイル運用では、画像などの実体ファイルは `app/assets/...` または `app/data/...` 配下に置き、問題データ側には参照パスを持たせる方針を検討する。

### 5.3 数式を含む問題

数学では、問題文や解説に数式が必要になる。

v0.7 時点では、まずは文字列として表現できる範囲から始める。必要に応じて、将来的に数式表示ライブラリの導入を検討する。

ただし、最初から数式表示の仕組みを作り込みすぎると、学校教科プロトタイプの範囲が広がりすぎる。そのため、最初の数学プロトタイプでは、単純な四則演算、正負の数、分数、小数など、プレーンテキストでも扱える問題から始める。

### 5.4 長文・会話文を含む問題

英語や国語では、長文、会話文、資料文を読ませたうえで答える問題が必要になる可能性がある。

この場合、問題文そのものと、参照する本文・会話文を分けて管理する方がよい。

たとえば、次のような考え方を検討する。

- passage: 長文や会話文
- question: 設問
- choices: 選択肢
- answer: 解答
- explanation: 解説

これにより、1つの長文に対して複数の設問を作る余地が生まれる。

### 5.5 複数資料を参照する問題

理科や社会では、複数の資料を見比べて答える問題が必要になる可能性がある。

たとえば、地図、年表、表、グラフ、模式図などを組み合わせる問題である。

v0.7 時点では、複数資料を本格的に扱う仕組みまでは作り込まない。ただし、将来拡張できるように、問題データの中で補助資料を配列として持てる形を検討する。

例:

```js
materials: [
  { type: "image", path: "assets/..." },
  { type: "table", title: "..." }
]
```

このようにしておけば、最初は1資料だけを扱い、必要になった段階で複数資料へ広げやすくなる。

### 5.6 v0.7で作り込みすぎないこと

v0.7 では、すべての問題形式に完全対応することを目指さない。

特に、次のような仕組みは、現時点では作り込みすぎない。

- 本格的なDBテーブル設計
- 問題管理画面
- 複雑な出題条件
- 高度な数式レンダリング
- 複数資料を自由に組み合わせる汎用ビューア
- ユーザー別の詳細な学習履歴管理

まずは、次の学校教科プロトタイプで必要になる最小限の表現を試し、実際に問題を作りながら拡張する。

## 6. 当面維持するもの

v0.7 では、問題データの分割や拡張形式を検討するが、現在うまく動いている仕組みを急に壊さないことを重視する。

当面は、次のものを維持する。

### 6.1 GitHub Pages で動く静的アプリ構成

現在の Junior Study Trainer は、GitHub Pages 上で `app/` 配下を公開する静的アプリとして動作している。

この構成は、サーバーやDBを必要とせず、すぐに動作確認できるというメリットがある。

v0.7 では、最終的なWebアプリ化・DB化を見据えつつも、当面はこの静的アプリ構成を維持する。

### 6.2 app/questions.js による実行方式

現行の `app/questions.js` 1ファイル実行方式は、Set001〜Set010 までの動作確認に使われており、すでに安定している。

そのため、v0.7 で問題データ分割を検討しても、すぐに `app/questions.js` を廃止しない。

必要に応じて、既存の `app/questions.js` を残しながら、`app/data/...` を使う方式を小さく検証する。

### 6.3 localStorage による進捗保存

現在の進捗保存は localStorage を使っている。

これは端末・ブラウザ・URLごとの保存であり、DBによるユーザー別管理ではない。

ただし、現段階では、個人利用のプロトタイプとしては十分に扱いやすい。そのため、v0.7 では localStorage を維持し、サーバー保存への移行は今後の検討とする。

### 6.4 小さく作って動かす進め方

このプロジェクトでは、最初から完成形を作るのではなく、まず小さく動くものを作り、実際に教材を増やしながら必要な設計を足していく。

v0.7 でもこの進め方を維持する。

特に、学校教科プロトタイプでは、最初からすべての問題形式に対応するのではなく、最小限の問題形式で動かし、必要になった段階で拡張する。

## 7. 今すぐ変更しないもの

v0.7 では、最終的なWebアプリ化・DB管理を見据えるが、今すぐ大きな構成変更は行わない。

ここでは、重要ではあるが、現時点では着手しないものを明確にする。

### 7.1 DB化

最終的には、問題データや学習履歴をDBで管理する構成を目指す。

しかし、v0.7 ではDB化そのものには着手しない。

DB化を始めると、データモデル、API、認証、管理画面、バックアップ、運用などの検討が必要になり、学校教科プロトタイプに進む前の範囲としては大きすぎるためである。

### 7.2 サーバーサイド実装

v0.7 では、サーバーサイドアプリケーションの実装は行わない。

現在はGitHub Pagesで動く静的アプリとして運用し、サーバーが必要になる機能は今後の段階で検討する。

### 7.3 ユーザー認証

ユーザーごとの進捗管理や複数端末同期を行うには、将来的にユーザー認証が必要になる可能性がある。

ただし、v0.7 では認証機能は扱わない。

当面は、個人利用を前提に、localStorage による端末単位の進捗保存を維持する。

### 7.4 管理画面

将来的には、問題を追加・編集・分類するための管理画面が必要になる可能性がある。

しかし、v0.7 では管理画面は作らない。

当面は、MarkdownやJavaScriptファイルを直接編集する方式で教材を管理する。

### 7.5 本格的な出題エンジン

将来的には、正答率、苦手分野、復習間隔、出題履歴などをもとに、出題順や復習タイミングを制御する仕組みが必要になる可能性がある。

ただし、v0.7 では本格的な出題エンジンは作らない。

まずは、教材データを安全に増やせる構造を整えることを優先する。

## 8. 学校教科プロトタイプで検証すること

v0.7 の次の段階では、学校教科プロトタイプを作成し、問題データ分割と拡張問題形式の妥当性を確認する。

最初の学校教科プロトタイプでは、理科を候補とする。

理由は、学校で使用している副教材と教科書をもとに、定期テスト対策としての到達度を段階的に設計しやすいためである。

### 8.1 最初の題材候補

最初の題材として、中学理科を扱う。

理科では、学校の副教材をベースにした基本問題と、教科書内容をもとにした発展問題を分けて作成できる。

まずは、副教材の内容をデータ化し、定期テストで80点程度を安定して狙えるレベルを目指す。

次に、副教材には載っていないが教科書内容から出題される可能性がある、やや難しめの問題を追加する。

この2段階をクリアすることで、定期テストで100点を目指せるレベルの教材構成にする。

### 8.2 難易度設計

学校教科プロトタイプでは、問題の難易度を次の2段階で整理する。

#### Level 1: 副教材ベース

副教材に載っている内容を中心に作成する。

このレベルは、定期テストで80点程度を安定して取ることを目的とする。

主に、次のような問題を扱う。

- 用語確認
- 基本知識
- 典型問題
- 授業・副教材で繰り返し扱われる内容
- 一問一答で確認しやすい内容

#### Level 2: 教科書ベースの発展・応用

副教材だけでは不足する可能性がある内容を、教科書から補う。

このレベルは、定期テストで100点を目指すことを目的とする。

主に、次のような問題を扱う。

- 教科書本文の細かい理解
- 図・表・グラフの読み取り
- 実験や観察の理由説明
- 用語の意味を説明する問題
- 複数の知識を組み合わせる問題
- 副教材にはないが、教科書から出題されそうな問題

### 8.3 検証したいデータ項目

理科プロトタイプでは、少なくとも次の項目を検証する。

- 問題ID
- 教科
- 学年
- 単元
- セットID
- 出典種別
- 難易度レベル
- 問題形式
- 問題文
- 選択肢
- 解答
- 解説
- タグ
- 表示形式
- 採点形式
- 補助資料の有無
- 補助資料の参照情報

特に、出典種別、難易度レベル、問題形式、補助資料の持ち方を検証することが重要である。

たとえば、次のような分類を想定する。

- source_type: workbook
- level: basic_80
- question_type: short_answer

また、教科書ベースの発展問題では、次のような分類を想定する。

- source_type: textbook
- level: advanced_100
- question_type: experiment_reasoning

これにより、副教材ベースの80点対策と、教科書ベースの100点対策を分けて管理できる。

### 8.4 PDF教材の扱い

学校教材はPDF化して参照できるようにする。

ただし、教科書や副教材のPDFには著作権がある可能性が高いため、公開リポジトリにそのまま格納することは避ける。

v0.7 時点では、PDF教材は次のように扱う方針とする。

- 教材PDFは、公開対象の app/ 配下には置かない
- GitHub Pagesで公開しない
- 必要に応じて、private repo またはローカル管理とする
- 問題データには、PDF本文そのものではなく、参照情報や自作した問題文を記録する
- 教科書・副教材の文章をそのまま大量に転載しない
- 問題は、学習目的に合わせて自作・要約・再構成する

教材PDFは、問題作成時の参照資料として扱う。

アプリで公開・表示する問題データには、必要最小限の情報だけを入れる。

### 8.5 問題形式は最初から網羅的に想定する

永続化方式やファイル分割方式は、DB化までの過渡期として扱う。

一方で、問題形式の取り込み方は、学校教科に進むこの段階で網羅的に想定する。

理由は、問題形式を後から場当たり的に追加すると、データ構造がその都度変わり、将来的なDB化や管理画面化が難しくなるためである。

最初の理科プロトタイプですべての問題形式を実装する必要はない。

しかし、データ構造を考える段階では、学校教科で必要になる問題形式を最初から受け止められるようにする。

特に理科では、次のような問題形式を想定する。

- 用語確認
- 一問一答
- 選択問題
- 短答問題
- 記述問題
- 図を見て答える問題
- 表を読んで答える問題
- グラフを読んで答える問題
- 実験・観察の理由を説明する問題
- 複数資料を参照する問題

v0.7 では、これらをすべて画面実装することを目的とはしない。

ただし、問題データの構造としては、次のような要素を持てるように検討する。

- 問題本文
- 補助資料
- 選択肢
- 解答
- 解説
- 出典種別
- 難易度レベル
- 問題形式
- 表示形式
- 採点形式
- タグ

この方針により、最初は副教材ベースの基本問題から始めつつ、後から教科書ベースの発展問題、図表問題、実験問題へ広げやすくする。

### 8.6 プロトタイプで確認すること

プロトタイプでは、次の点を確認する。

- 副教材ベースの問題をセット別ファイルで管理しやすいか
- 出典種別を持たせることで、80点対策と100点対策を分けられるか
- 難易度レベルを持たせることで、基本問題と発展問題を分けられるか
- 理科の用語確認・基本知識・理由説明に対応できるか
- 図・表・グラフ・実験問題を将来追加できるデータ構造になっているか
- 問題形式を追加しても、既存データを大きく壊さずに済むか
- 将来DB化するときに移行しやすい粒度になっているか

この検証により、学校教科データを増やす前に、過渡期のファイル分割と問題データ表現の妥当性を確認する。
